# SOLID 代码异味提示

## SRP（单一职责原则）

- 文件包含不相关的多个关注点（如 HTTP + 数据库 + 业务规则混在一个文件）
- 类/模块过大，内聚性低，或存在多个变更原因
- 函数编排了许多不相关的步骤
- 上帝对象，对系统了解过多
- **自问**："这个模块唯一的变更原因是什么？"

## OCP（开闭原则）

- 添加新行为需要修改大量 switch/if 分支
- 功能增长需要修改核心逻辑而非扩展
- 没有插件/策略/钩子扩展点
- **自问**："我能在不修改现有代码的情况下添加新变体吗？"

## LSP（里氏替换原则）

- 子类检查具体类型或对基类方法抛出异常
- 重写的方法削弱了前置条件或加强了后置条件
- 子类忽略或空实现父类行为
- **自问**："我能在调用方不知情的情况下替换任意子类吗？"

## ISP（接口隔离原则）

- 接口方法过多，实现方大多未使用
- 调用方依赖宽泛接口只为满足少量需求
- 接口方法存在空实现或桩实现
- **自问**："所有实现方都用到了所有方法吗？"

## DIP（依赖倒置原则）

- 高层逻辑依赖具体的 IO、存储或网络类型
- 硬编码实现而非抽象或注入
- 导入链将业务逻辑与基础设施耦合
- **自问**："我能在不修改业务逻辑的情况下替换实现吗？"

---

## 常见代码异味（超出 SOLID 范畴）

| 异味 | 表现 |
|------|------|
| **过长方法** | 函数超过 30 行，嵌套层级过多 |
| **依恋情结** | 方法使用其他类的数据多于自身数据 |
| **数据泥团** | 相同的一组参数反复一起传递 |
| **基本类型偏执** | 用字符串/数字代替领域类型 |
| **霰弹式修改** | 一处变更需要修改多个文件 |
| **发散式变化** | 一个文件因多个不相关原因而变更 |
| **死代码** | 不可达或从未调用的代码 |
| **过度设计** | 为假想的未来需求创建抽象 |
| **魔法数字/字符串** | 硬编码值没有命名常量 |

## 重构启发式方法

1. **按职责拆分，而非按大小** - 小文件也可能违反 SRP
2. **仅在需要时引入抽象** - 等到第二个使用场景出现
3. **保持重构渐进式** - 移动前先隔离行为
4. **先保证行为正确** - 重构前先加测试
5. **按意图命名** - 命名困难说明抽象可能有问题
6. **优先组合而非继承** - 继承会产生紧耦合
7. **让非法状态无法表示** - 用类型系统强制约束不变量
