# 安全与可靠性清单

## 输入/输出安全

- **XSS**：不安全的 HTML 注入、`dangerouslySetInnerHTML`、未转义的模板、innerHTML 赋值
- **注入**：通过字符串拼接或模板字面量导致的 SQL/NoSQL/命令/GraphQL 注入
- **SSRF**：用户可控的 URL 在未经白名单验证的情况下访问内部服务
- **路径遍历**：用户输入未经净化直接用于文件路径（`../` 攻击）
- **原型污染**：JavaScript 中不安全的对象合并（`Object.assign`、使用用户输入的展开运算符）

## 认证/授权

- 读写操作缺少租户或所有权检查
- 新接口没有认证守卫或 RBAC 强制执行
- 信任客户端提供的角色/标志/ID
- 访问控制失效（IDOR - 不安全的直接对象引用）
- 会话固定或弱会话管理

## JWT 与令牌安全

- 算法混淆攻击（接受 `none` 或 `HS256`，而预期是 `RS256`）
- 弱密钥或硬编码密钥
- 缺少过期时间（`exp`）或未验证
- JWT 载荷中包含敏感数据（令牌是 base64 编码，非加密）
- 未验证 `iss`（签发方）或 `aud`（受众）

## 密钥与个人信息

- 代码/配置/日志中出现 API Key、令牌或凭据
- 密钥在 git 历史中或环境变量暴露给客户端
- 日志中过度记录个人信息或敏感载荷
- 错误信息中缺少数据脱敏

## 供应链与依赖

- 未固定版本的依赖，可能被恶意更新
- 依赖混淆（私有包名冲突）
- 从不可信来源或无完整性校验的 CDN 导入
- 存在已知 CVE 的过时依赖

## CORS 与响应头

- 过于宽松的 CORS（`Access-Control-Allow-Origin: *` 同时携带凭据）
- 缺少安全响应头（CSP、X-Frame-Options、X-Content-Type-Options）
- 暴露内部响应头或堆栈跟踪

## 运行时风险

- 无界循环、递归调用或大型内存缓冲区
- 外部调用缺少超时、重试或限流
- 请求路径上的阻塞操作（异步上下文中的同步 I/O）
- 资源耗尽（文件句柄、连接、内存）
- ReDoS（正则表达式拒绝服务）

## 密码学

- 弱算法（MD5、SHA1 用于安全目的）
- 硬编码的 IV 或盐值
- 使用无认证的加密（ECB 模式、无 HMAC）
- 密钥长度不足

## 竞态条件

竞态条件是导致间歇性故障和安全漏洞的隐蔽 bug，需重点关注：

### 共享状态访问
- 多线程/协程/异步任务在无同步的情况下访问共享变量
- 全局状态或单例被并发修改
- 懒初始化缺少适当的锁（双重检查锁定问题）
- 在并发上下文中使用非线程安全的集合

### 先检查后操作（TOCTOU）
- `if (exists) then use` 模式缺少原子操作
- `if (authorized) then perform` 中授权状态可能在操作间改变
- 文件存在性检查后再执行文件操作
- 余额检查后再扣款（金融操作）
- 库存检查后再下单

### 数据库并发
- 缺少乐观锁（`version` 列、`updated_at` 检查）
- 缺少悲观锁（`SELECT FOR UPDATE`）
- 无事务隔离的读-改-写操作
- 计数器递增缺少原子操作（`UPDATE SET count = count + 1`）
- 并发插入导致唯一约束冲突

### 分布式系统
- 共享资源缺少分布式锁
- 主节点选举竞态条件
- 缓存失效竞争（写入后读到旧数据）
- 事件顺序依赖缺少适当的排序机制
- 集群操作中的脑裂场景

### 需标记的常见模式
```python
# 危险模式：
if not exists(key):       # TOCTOU
    create(key)

value = get(key)          # 读-改-写
value += 1
set(key, value)

if user.balance >= amount:  # 先检查后操作
    user.balance -= amount
```

### 自问
- "如果两个请求同时执行这段代码会怎样？"
- "这个操作是原子的，还是可能被中断？"
- "这段代码访问了哪些共享状态？"
- "在高并发下这段代码的行为如何？"

## 数据完整性

- 缺少事务、部分写入或不一致的状态更新
- 持久化前验证不足（类型强制转换问题）
- 可重试操作缺少幂等性
- 并发修改导致更新丢失
