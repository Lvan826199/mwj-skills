## 各章节写作指引

### 章节 1：模块概述

- **1.1 模块简介**：一段话描述模块的定位和核心职责
- **1.2 业务目标**：4~5 条 bullet list，描述模块要达成的业务目标
- **1.3 适用范围**：一段话描述目标用户和使用场景
- **1.4 核心概念**（可选）：当模块有专有术语时，用表格解释

示例（核心概念表格）：

| 概念 | 说明 |
|------|------|
| 测试套件 | 一组测试用例的集合，可统一执行 |
| 数据驱动 | 使用外部数据源驱动测试用例执行 |

### 章节 2：功能需求

这是文档的核心章节。每个功能占一个子节（2.1, 2.2, 2.3...），内部结构统一：

**功能描述**：一段话说明功能做什么

**业务规则**（2 列表格）：

| 规则项 | 说明 |
|--------|------|
| 名称唯一 | 同一项目内名称不可重复 |
| 状态流转 | 新建 → 进行中 → 已完成 |

**输入项/配置项**（4 列表格）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 名称，最大 100 字符 |
| description | string | 否 | 描述信息 |
| priority | string | 是 | 优先级：高/中/低 |

**输出结果**（视情况，用 JSON 示例或文字描述）：

```json
{
  "id": "记录ID",
  "name": "名称",
  "status": "状态",
  "created_at": "创建时间"
}
```

**操作列表**（2 列表格，视情况）：

| 操作 | 说明 |
|------|------|
| 创建 | 创建新记录 |
| 编辑 | 修改已有记录 |
| 删除 | 删除记录（需二次确认） |

**验收标准**（2 列表格，每个功能必须包含）：

列出该功能的关键验收场景，覆盖正常流程和异常流程：

| 场景 | 预期结果 |
|------|----------|
| 正常创建（所有必填项填写正确） | 返回新记录ID，列表中可见新记录 |
| 名称重复 | 返回 409 错误，提示"名称已存在" |
| 必填项缺失 | 返回 422 错误，提示具体缺失字段 |
| 无权限操作 | 返回 403 错误，提示"无权限执行此操作" |

### 章节 3：非功能需求

三个固定子节，每个用 2 列表格：

**性能要求**（指标值不要直接套用模板默认值，应根据实际业务场景填写；如果用户未提供，标注 `{待确认}` 并在阶段一完成时提醒用户补充）：

| 指标 | 要求 |
|------|------|
| 列表加载 | {待确认，建议 < 500ms} |
| 搜索响应 | {待确认，建议 < 1s} |
| 批量操作 | {待确认，视数据量级} |

**安全要求**：

| 项目 | 要求 |
|------|------|
| 数据加密 | 敏感数据使用 AES-256 加密 |
| 权限控制 | 基于角色的访问控制 |

**可用性要求**：

| 项目 | 要求 |
|------|------|
| 容错处理 | 操作失败时给出明确提示 |
| 数据备份 | 支持数据导出和恢复 |

### 章节 4：业务流程

每个流程用纯文本箭头格式，放在代码块中：

简单顺序流程：
```
用户进入列表页
    ↓
点击创建按钮
    ↓
填写表单信息
    ↓
提交表单
    ↓
系统验证数据
    ├─→ 验证通过：保存数据，返回列表
    └─→ 验证失败：提示错误信息
```

多级分支流程：
```
用户提交Bug
    ↓
系统自动分配
    ↓
开发人员确认
    ├─→ 确认修复：标记为修复中
    │       ↓
    │   提交修复代码
    │       ↓
    │   测试人员验证
    │       ├─→ 验证通过：关闭Bug
    │       └─→ 验证失败：重新打开
    └─→ 非Bug：标记为已拒绝
```

### 章节 5：数据约束

- **5.1 唯一性约束**：bullet list 列出唯一性要求
- **5.2 数据长度约束**（3 列表格）：

| 字段 | 最大长度 | 说明 |
|------|----------|------|
| 名称 | 100 | name 字段 |
| 描述 | 500 | description 字段 |

- **5.3 数据格式/枚举约束**：bullet list 或表格描述

### 章节 6：界面原型要点

每个页面一个子节，用 bullet list 描述布局和交互要点：

**页面布局：**
- 顶部：搜索栏 + 筛选条件
- 中部：数据表格，支持排序和分页
- 操作列：编辑、删除、查看详情
- 右上角：新建按钮

**交互状态（每个页面必须覆盖）：**
- 空状态：列表无数据时的展示（如"暂无数据，点击新建"）
- 加载状态：数据加载中的展示方式（骨架屏 / loading 动画）
- 错误状态：接口失败时的提示方式（toast / 内联错误）
- 操作反馈：增删改成功后的行为（刷新列表 / 跳转 / toast 提示）

### 章节 7：扩展需求

每个扩展功能一个子节，用 2~3 条 bullet list 描述未来可能的功能扩展。

## 格式规范

### 语言
- 全文使用简体中文

### 章节分隔
- 每个一级章节（## 1. ~ ## 7.）之间用 `---` 分隔线隔开

### 表格格式汇总

| 使用场景 | 列定义 |
|----------|--------|
| 业务规则 | `| 规则项 | 说明 |` |
| 输入项/配置项 | `| 字段 | 类型 | 必填 | 说明 |` |
| 操作列表 | `| 操作 | 说明 |` |
| 验收标准 | `| 场景 | 预期结果 |` |
| 性能/安全/可用性 | `| 指标/项目 | 要求 |` |
| 数据长度约束 | `| 字段 | 最大长度 | 说明 |` |
| 核心概念 | `| 概念 | 说明 |` |
| 枚举约束 | `| 字段 | 可选值 | 说明 |` |

### 流程图
- 使用纯文本箭头格式，放在 ``` 代码块中
- 用 `↓` 表示顺序流转，缩进 4 空格
- 分支用 `├─→` 和 `└─→` 表示
- 多级分支用 `│` 连接竖线对齐

### JSON 示例
- 使用 ```json 代码块
- 字段名使用英文 snake_case
- 值使用中文描述

### 标题层级
- `#` — 文档标题（仅一个）
- `##` — 一级章节（1~7）
- `###` — 二级章节（如 2.1, 2.2）
- `####` — 三级章节（如 2.1.1, 2.1.2）

### 命名规范
- 文件名：`{编号}-{模块名称}.md`（如 `12-通知管理模块.md`）
- 字段名（JSON/表格）：英文 snake_case
- 枚举值：中文描述
