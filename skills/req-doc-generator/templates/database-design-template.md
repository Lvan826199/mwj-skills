# 数据库设计文档模板

本文件定义数据库设计文档的标准结构和写作指引。生成时必须严格遵循此模板，并与项目已有的 `doc/数据库设计.md` 格式保持一致。

---

## 输出位置

将新模块的数据库设计**追加**到 `doc/数据库设计.md` 文件末尾，使用与已有模块一致的格式。

---

## 字段定义表规范

### 表头固定为 6 列

| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 说明 |

### 必备基础字段（每张表必须包含）

| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 说明 |
|--------|------|------|----------|--------|------|
| id | BIGINT | - | NO | - | 主键ID，自增 |
| created_at | DATETIME | - | NO | CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | - | NO | CURRENT_TIMESTAMP ON UPDATE | 更新时间 |

### 软删除字段（业务实体表必须包含）

| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 说明 |
|--------|------|------|----------|--------|------|
| deleted_at | DATETIME | - | YES | NULL | 软删除时间 |

### 创建人/更新人字段（按需包含）

| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 说明 |
|--------|------|------|----------|--------|------|
| created_by | BIGINT | - | YES | NULL | 创建人ID |
| updated_by | BIGINT | - | YES | NULL | 更新人ID |

## 常用字段类型参考

| 场景 | 类型 | 长度 | 示例 |
|------|------|------|------|
| 主键 | BIGINT | - | id |
| 短文本（名称） | VARCHAR | 50~100 | name, title |
| 中文本（描述） | VARCHAR | 200~500 | description |
| 长文本（内容） | TEXT | - | content, body |
| 枚举/状态 | ENUM 或 VARCHAR | 20~50 | status, type |
| 布尔值 | BOOLEAN | - | is_active, is_public |
| 整数 | INT | - | sort_order, count |
| 大整数/外键 | BIGINT | - | user_id, project_id |
| 时间 | DATETIME | - | created_at |
| 日期 | DATE | - | start_date |
| JSON 数据 | JSON | - | config, metadata |
| URL/路径 | VARCHAR | 255~500 | avatar, file_path |
| IP 地址 | VARCHAR | 50 | ip_address |

## 索引命名规范

| 索引类型 | 命名格式 | 示例 |
|----------|----------|------|
| 主键 | PRIMARY | PRIMARY |
| 唯一索引 | uk_{字段名} | uk_username, uk_email |
| 联合唯一索引 | uk_{表简称}_{字段1}_{字段2} | uk_user_role |
| 普通索引 | idx_{字段名} | idx_status, idx_created_at |
| 联合索引 | idx_{字段1}_{字段2} | idx_project_module |

---

## 外键命名规范

| 格式 | 示例 |
|------|------|
| fk_{当前表简称}_{关联表简称} | fk_user_role_user |
| fk_{当前表}_{关联字段} | fk_bug_project_id |

### 删除规则选择

| 规则 | 适用场景 |
|------|----------|
| CASCADE | 关联表数据随主表删除（如：用户角色关联） |
| SET NULL | 主表删除后关联字段置空（如：操作日志的用户ID） |
| RESTRICT | 禁止删除有关联数据的主表记录（如：有用例的项目） |

## 表名前缀规范

根据模块归属选择表名前缀，与项目已有表保持一致：

| 前缀 | 适用模块 | 示例 |
|------|----------|------|
| sys_ | 系统管理、用户权限 | sys_user, sys_role |
| proj_ | 项目管理 | proj_project, proj_module |
| test_ | 功能测试 | test_case, test_execution |
| api_ | 接口测试 | api_definition, api_case |
| web_ | Web 自动化 | web_page, web_element |
| app_ | App 自动化 | app_device, app_case |
| bug_ | 缺陷管理 | bug_record, bug_history |
| rpt_ | 测试报告 | rpt_report, rpt_analysis |

> 注意：如果项目中已有该模块的表，必须沿用已有前缀，不要自创新前缀。生成前先检查 `doc/数据库设计.md` 中已有的表名。

## 设计原则

1. **从需求文档提取实体**：需求文档中每个核心功能涉及的数据实体都应有对应的表
2. **关联表独立建表**：多对多关系必须建立中间关联表
3. **字段与需求一致**：需求文档中定义的输入项/配置项字段必须在表中体现
4. **枚举值一致**：需求文档中定义的枚举值必须与数据库 ENUM 或约束一致
5. **冗余字段标注**：如果为查询性能添加冗余字段，在说明中标注"冗余"
6. **预置数据完整**：系统预定义的枚举、角色、权限等必须列出预置数据
