# 模块设计文档生成流水线

## 目标

根据用户提供的模块信息，按照 **需求文档 → 数据库设计 → API 设计** 三阶段流水线，逐步生成完整的模块设计文档。每个阶段的输出是下一阶段的输入依据。

生成的文档是专业的软件设计文档，适用于开发团队作为实现依据。

---

## 流水线总览

```
阶段一：需求文档
    ↓ （用户确认后）
阶段二：数据库设计
    ↓ （用户确认后）
阶段三：API 设计文档
```

每个阶段完成后，必须等待用户确认再进入下一阶段。用户可以在任何阶段要求修改。

---

## 阶段零：信息收集

在开始生成前，先向用户确认以下关键信息：

1. **模块名称** — 如"通知管理模块"、"订单管理模块"
2. **业务背景** — 这个模块解决什么问题，属于什么类型的系统
3. **核心功能列表** — 模块包含哪些主要功能
4. **目标用户** — 谁会使用这个模块
5. **技术约束**（可选） — 是否有现有表需要兼容？使用什么数据库？是否有特定框架限制？
6. **数据量级预估**（可选） — 日均新增记录数、预期并发量，影响索引策略和分页设计

如果用户已提供足够信息（模块名称 + 至少 2 个核心功能），可直接开始。信息明显不足时先提问收集。第 5、6 项为可选，用户未提供时可跳过，但如果用户提供了则必须在数据库设计和 API 设计阶段体现。

---

## 阶段一：需求文档

### 输入
- 用户提供的模块信息

### 模板
- 严格按照 `templates/requirement-template.md` 中定义的 7 大章节结构和写作指引生成

### 生成原则
- **不确定的业务细节必须向用户确认**，不要自行假设（如：具体字段定义、枚举值范围、性能指标阈值、权限规则）
- **可以合理推断的内容直接生成**（如：基于功能描述推断的业务规则、常见非功能需求指标）
- **非功能需求中的性能指标不要硬编码默认值**，如果用户未在阶段零提供数据量级信息，性能指标标注为 `{待确认}` 并在阶段完成时提醒用户补充
- **每个功能点必须包含验收标准**，覆盖正常流程和至少 2 个异常场景（如参数校验失败、权限不足、数据冲突等）
- 复杂模块（超过 5 个核心功能）建议分批生成，先输出章节 1~3，确认后再输出 4~7

### 输出
- 保存为 `doc/平台详细需求文档/{编号}-{模块名称}.md`
- 编号根据已有文档顺序递增

### 阶段完成
- 输出完成后告知用户："需求文档已生成，请确认内容。确认后我将基于此生成数据库设计文档。"

---

## 阶段二：数据库设计

### 输入
- 阶段一生成的需求文档（实体、字段、关系、约束）

### 模板
- 严格按照 `templates/database-design-template.md` 中定义的表结构格式生成

### 生成原则
- 从需求文档中提取所有实体和关系，转化为数据库表
- 每张表必须包含：字段定义表、索引、外键
- 遵循项目已有的数据库设计风格（参考 `doc/数据库设计.md` 中已有模块的格式）
- 表名使用项目统一前缀规则（如 `sys_`、`test_`、`api_` 等，根据模块归属决定）
- 所有表必须包含 `id`、`created_at`、`updated_at` 基础字段
- 支持软删除的表必须包含 `deleted_at` 字段
- 关联表必须定义外键和级联规则

### 输出
- 将新模块的数据库设计追加到 `doc/数据库设计.md` 文件末尾
- 模块编号与需求文档保持一致

### 阶段完成
- 输出完成后告知用户："数据库设计已生成，请确认表结构。确认后我将基于需求文档和数据库设计生成 API 设计文档。"

---

## 阶段三：API 设计文档

### 输入
- 阶段一的需求文档（功能列表、业务规则、输入输出）
- 阶段二的数据库设计（数据结构、字段类型）

### 模板
- 严格按照 `templates/api-design-template.md` 中定义的接口文档格式生成

### 生成原则
- 从需求文档的每个功能点推导出对应的 API 接口
- 请求/响应字段必须与数据库设计保持一致
- 遵循项目已有的 API 设计风格（参考 `doc/平台后端API设计文档/` 中已有模块的格式）
- 使用统一响应格式 `ResponseModel[T]`
- 所有列表接口必须支持分页
- 所有接口必须定义错误码和错误场景

### 输出
- 保存为 `doc/平台后端API设计文档/{编号}-{模块名称}API.md`
- 编号与需求文档保持一致

### 阶段完成
- 输出完成后告知用户："API 设计文档已生成。三阶段文档流水线全部完成。"
- 汇总输出的文件列表

---

## 通用规则

### 语言
- 全文使用简体中文

### 文件命名
- 需求文档：`{编号}-{模块名称}.md`（如 `12-通知管理模块.md`）
- API 文档：`{编号}-{模块名称}API.md`（如 `12-通知管理模块API.md`）
- 数据库设计：追加到 `doc/数据库设计.md`

### 一致性
- 三份文档中的实体名称、字段名称、枚举值必须完全一致
- API 的请求/响应字段必须与数据库字段对应（允许合理的字段映射，如 `password_hash` 不暴露给 API）
- 需求文档中的业务规则必须在 API 文档中体现为验证逻辑

### 字段命名
- 数据库字段：英文 snake_case
- API 字段：英文 snake_case
- 枚举值：英文小写（如 `active`、`disabled`）

### 用户可选择的执行范围
- 用户可以要求只执行某个阶段（如"只生成数据库设计"）
- 此时跳过其他阶段，但仍需读取已有的上游文档作为输入
- 例如：只生成 API 文档时，需先读取已有的需求文档和数据库设计
